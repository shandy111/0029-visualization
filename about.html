Methodology & data sources

How this website was built

This page records both the shared project workflow and the individual methodology contributions of group members. The aim is to show how the project moved from processed charging and Census data to a final web-based narrative, while also making each person's role clear.

Project design

Analytical logic from raw data to final story

Research question

The project asks where London's public EV charging infrastructure is concentrated, how evenly it is distributed across neighbourhoods, and how LSOA-level inequality translates into borough-level rollout differences.

Why LSOA first

LSOAs are the main unit of spatial analysis because borough averages can hide fine-grained access gaps. We therefore begin with neighbourhood-scale charger presence and density before moving upward to Inner/Outer and borough summaries.

Interpretive framework

Deprivation, car ownership, housing type and occupation are treated as contextual indicators rather than single-variable explanations. The website is designed as a data story that compares multiple dimensions of inequality instead of relying on one map alone.

Indicator definitions

Key indicators used in the website

Chargers per 10,000 residents

A population-adjusted density indicator used across LSOA, borough and Inner/Outer London summaries.

Zero-charger LSOA

A simple access flag showing whether a neighbourhood has no recorded public charger at all.

Chargers per 1,000 car owning households

A demand-context indicator comparing public charging provision with the number of car-owning households.

Percentage(%) car owning households

Derived from Census 2021 TS045. Used as a demand-context proxy rather than direct evidence of EV ownership.

Percentage(%) flats / apartments

Derived from Census 2021 TS044. Used as a proxy for potential home-charging constraints rather than direct evidence of parking conditions.

IMD Decile

Allows the analysis to compare public EV charging provision across more deprived and less deprived LSOAs.

Team contributions

Methodology

Member 1: Data integration and metric development

Before making visual charts, the project cleaned up and integrated multiple data sets, and finally formed a spatial data set based on the LSOA hierarchy. The main data sources include: publicly available JSON format electric vehicle charging point data, 2021 LSOA boundary data, population data, IMD cental, census variables on housing property rights, occupation and car ownership, and PTAL data. These data sets are selected to reflect both the distribution of charging facilities and the community characteristics that may help explain spatial inequality. First, clean up the charging pile data and eliminate the missing or invalid records of latitude and longitude values. Then convert the remaining records into spatial point data in the R environment. Then, the coordinate reference system of the charging pile and the LSOA boundary is standardised, so that the charging pile and the LSOA polygon can be spatially associated through the "point in the polygon" operation (`st_within`). This process derives the number of charging piles in each LSOA. Subsequently, the original number of charging points was standardised using population data. The main indicator is the number of charging points per 10,000 residents, as well as the total number of charging points and a binary indicator (used to show whether there are public charging points in LSOA). The data also contains the name of the administrative district and the classification information of London's inner ring/outer ring for comparison on different spatial scales. Other variables (including material scarcity index (IMD), housing property type, occupational structure, private car ownership rate and public transportation accessibility level (PTAL)) are all associated at the LSOA level. Housing and occupation variables are reorganised into broader categories and converted into percentages for a more consistent comparison of LSOA. Finally, all processed variables are exported as a unified GeoJSON file for use in Leaflet maps and Chart.js charts. In addition, district-level summary statistics are also generated to promote broader comparisons within London.

Member 2: Spatial Visualisation and Thematic Explanation Modules

The main spatial visualisation—a choropleth map—displays charger density at the LSOA level. LSOA polygons from the 2021 boundary dataset were converted to GeoJSON format using R packages and utilised as a vector file. Charger density was categorised using fixed breakpoints (0, 1–5, 5–15, 15–40, 40+) and represented using a sequential blue colour scheme. An orange dotted line delineating Inner London was generated by processing all Inner London LSOA polygons in R and simplifying the geometry to reduce file size. Hover interactions display LSOA-level attributes, including borough, inner/outer classification, number of chargers, chargers per 10,000 residents, and IMD decile. On the basis of spatial distribution, this study combines existing literature (Zhang et al., 2024; Lou et al., 2024; Ermagun & Tian, 2024) to select housing structure, occupational composition and public transportation accessibility (PTA L) As an explanatory variable, and build the corresponding thematic visualisation modules separately. The housing module uses a switchable Choropleth map to show the proportional distribution of different property types at the LSOA level, and embeds three types of charts in the sidebar: the scatter plot diagram draws the relationship between housing ownership and charger density based on LSOA data, and calculates the Pearson correlation coefficient at the front end; The grouping bar chart compares the housing structure of different regions by calculating the average value of inner/outer classification; the quartile chart is grouped by sorting the ownership rate and calculating the average charging density of each group to achieve hierarchical comparison. The occupation module adopts five types of NS-SeC grouping, displays the proportion of different occupations through the field switching control map, and screens the LSOA of the top 25% of each occupation in the chart to calculate the average charging density, while grouping and comparing internal and external London. The PTAL module uses the Choropleth map of sequential colour matching to express the accessibility level, and combines the scatter map, the hierarchical grouping bar chart and the interval distribution map to visualise the expression from the three levels of continuous variable relationship, graded aggregation and spatial structure. All charts are implemented through Chart.js and share the same data structure with the map to realise linkage updates through variable switching.

Yu Shi: borough and contextual comparison workflow

My contribution focused on connecting processed spatial and socio-economic data to the final web-based visualisation of EV charging inequality in London. I worked mainly on the borough-level and contextual comparison components, and on integrating these outputs into the project website.

I started from the existing lsoa_ev.geojson dataset rather than rebuilding the full charger geography from scratch. In process_demand_indicators.py, I cleaned Census 2021 TS045 and TS044 tables, standardised category fields, converted counts into comparable indicators, and merged them into lsoa_ev_plus_demand.geojson. I then used process_borough_summary.py to validate borough summary fields, derive indicators such as pct_no_charger, and generate London-wide metrics and chart-ready borough outputs. To extend borough context, I used process_ts044_context.py to aggregate car ownership and housing indicators at borough level. On the front end, I integrated these outputs into the homepage and methodology page, adjusted layout, navigation and copy width, and refined Chart.js interactions so the borough and comparison sections were clearer and more consistent.

Web production

How processed data became the final website

Interactive map layer

map.html serves as the main spatial entry point. It uses the processed LSOA GeoJSON to show neighbourhood-scale inequality, allowing the website to open with a strong geographical overview before moving into chart-based interpretation.

Comparative chart layer

Chart.js is used to build comparison charts including IMD summaries, Inner/Outer London comparisons, borough demand scatter plots, housing context views and borough ranking charts. These chart sections read processed CSV outputs directly in the browser.

Narrative page assembly

The homepage was assembled as a single scrollytelling-style page that moves from map, to coverage gap, to Inner/Outer comparison, to deprivation, demand, housing, borough analysis and final discussion. The methodology page documents this pipeline so the website can be read both as a visual story and as a reproducible analytical project.

Limitations

Key limitations

  • OpenChargeMap is an open and partly crowdsourced proxy source, so charger coverage may be incomplete.
  • The project analyses public chargers, not private chargers.
  • Car ownership is not EV ownership.
  • Housing type is not direct evidence of parking availability.
  • LSOA aggregation can still hide very local variation within neighbourhoods.
  • Borough-level rollout patterns are observed, but policy causality is not proven.
